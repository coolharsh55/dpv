1.1. CSV Schema Guidelines

  • Term
    • the identifier for the concept that will be suffixed to the namespace to create its unique identify e.g. PersonalData with namespace dpv results in the concept dpv:PersonalData
    • the identifier should following naming conventions as PascalCase for classes and camelCase for properties
    • there should be no spaces, dashes, underscores
    • acronyms should be retained in ALLCAPS
    • plurals should not be used unless intentional and required
    • property terms should have a prefix e.g. of the form has,is,for to associate a concept and indicate its applicability
    • specialising a concept should follow the naming conventions of the term being specialised
    • spellings should use UK english unless intentional and required to use an alternate form
    • terms should be both machine and human readable
    • Referencing External Terms
      • where external terms are to be used, e.g. to describe an existing concept or a property being reused to provide contextual association, the Term should be of the form prefix:Term with the appropriate namespace defined in DPV Namespace spreadsheet
      • the external term should have a label, a description, and a status. No other information should be provided within the table
  • Label
    • descriptive labels for human consumption
    • may differ from Term
    • must follow the same naming convention as the term, with additional punctuations where necessary, e.g. Pascal Case for classes and camel case (all lower) for properties
  • Description
    • a definition of the term describing its use and role within DPV
    • should be phrased in context of the concept being applied, e.g. for classes it should describe the information represented by the concept and for properties it should describe the association, relation, or annotation
  • ParentTerm and ParentType
    • together these describe how the Term will be defined
    • a represents rdf:type instance of the ParentTerm
    • sc represents rdfs:subClassOf of the ParentTerm
    • use of prefix:Topconcept in ParentType means Term a Topconcept
    • if topconcept is present, then stating ParentType means the Term is an instance of the topconcept and has skos:broader ParentTerm
  • examples
    • Term is Purpose, ParentType is a, and ParentTerm is rdfs:Class
      • Purpose a rdfs:Class
    • Term is SpecialPurpose, ParentType is sc, and ParentTerm is dpv:Purpose
      • SpecialPurpose rdfs:subClassOf dpv:Purpose
    • Term is Marketing, ParentType is dpv:Purpose, and ParentTerm is Empty
      • Marketing a dpv:Purpose
      • Marketing skos:broader dpv:Purpose
    • Term is DirectMarketing, ParentType dpv:Purpose, ParentTerm is dpv:Marketing
      • DirectMarketing a dpv:Purpose
      • DirectMarketing skos:broader dpv:Marketing

6. DONE RDFORM phasing out

  • [X] determine whether RDFORM should be phased out (yes - it seems). The purpose of the ORM is to make it easy to pull related properties and concepts using python classes/dicts during RDF generation and HTML generation. However, it is buggy and prone to causing cryptic issues. A better simpler structure might be to keep a global list and to use functional wrappers to get the information from it using rdf prefixes, and optionally to have an API to get the specific forms of information e.g. strings, IRIs, list of parent classes, list of subclasses, etc. - which are needed in the documentation. Also needed are prefixed forms e.g. "dpv-pd:Email" from the IRI
